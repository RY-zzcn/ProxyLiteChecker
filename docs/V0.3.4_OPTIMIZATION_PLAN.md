# ProxyLiteChecker v0.3.4 优化实施计划

状态：已完成

目标版本：`v0.3.4`

制定日期：2026-07-10
主题：目标可用性、任务终态、取消安全和代理生命周期可靠性

## 1. 背景

`v0.3.3` 已具备多目标检测、目标化 HTTP/SOCKS5 网关、失败隔离、运行时网关配置和 GeoIP 国家过滤。但当前任务与数据模型仍保留部分早期单目标语义，主要表现为：

- 命名目标只要基础出口 IP 可达，就会被标记为目标可用并进入目标网关。
- `proxies` 主表状态会被最后一次目标检测覆盖，与 `proxy_checks` 的目标状态形成两套真相。
- 开启“失败即删”时，一个目标失败可能删除在其它目标仍然可用的代理。
- 取消请求会立即释放重任务冲突锁，旧任务尚未退出时新任务可能启动。
- 拉取任务即使全部代理源失败，前后端仍可能显示“任务已完成”。
- 可用转待检后，待检删除仍按旧检测时间计算，待检代理可能得不到完整 TTL。

本版本定位为可靠性热修，不引入多节点、分布式队列或新的大型依赖。

## 2. 本版本目标

### 2.1 目标可用性

- `generic` 目标允许基础出口或通用站点可达时判定可用。
- `openai`、`grok`、`gemini`、`claude` 必须满足对应 Web 或 API 至少一项可达，才能判定目标可用。
- 仅基础出口可达时保留 `recommended_use=base`，但不得进入命名目标网关和导出池。
- 启动迁移会修正旧版本中已经保存的“命名目标 base-only 可用”记录。

### 2.2 单一聚合状态

- `proxy_checks` 继续作为目标维度状态的权威来源。
- `proxies.status` 改为目标状态的安全聚合：任一目标可用则全局可用；没有可用目标时再区分待检和失败。
- 保存目标检测结果后同步单个代理的聚合状态。
- 启动迁移后同步现有代理的聚合状态，避免最后检测目标覆盖全局状态。

### 2.3 安全失败删除

- 检测过程中不再因为单个目标失败立即删除代理。
- 等同一代理在本次计划中的所有目标完成后，再判断是否允许删除。
- 只有所有本次目标均失败、没有基础出口能力且结果已成功持久化时，才执行“失败即删”。
- 自动维护只清理有 `generic` 基础失败证据且不存在任何目标可用结果的代理。

### 2.4 任务状态机

任务状态统一为：

```text
running
  -> cancel_requested
  -> cancelled

running
  -> completed
  -> partial
  -> failed
```

规则：

- `cancel_requested` 仍属于活动任务，继续阻止新的拉取或检测重任务。
- worker 完全退出后才能进入 `cancelled`。
- 全部代理源失败时任务进入 `failed`。
- 部分代理源失败或部分检测结果持久化失败时进入 `partial`。
- goroutine panic 必须转成任务失败，不允许任务永久停留在活动状态。
- 内存任务历史保留上限，避免长期运行后无限增长。

### 2.5 前端终态

- 手动任务开始后持续查询具体 `job_id`，不再仅依赖活动任务列表。
- 根据 `completed`、`partial`、`failed`、`cancelled` 显示不同提示。
- 取消后继续显示“正在停止”，直到后端确认终态。
- 使用串行 `setTimeout` 轮询，避免慢请求导致多个 `setInterval` 周期重叠。

### 2.6 生命周期和调度

- 新增 `status_changed_at`，明确记录进入当前状态的时间。
- 待检过期删除按进入待检状态的时间计算，不再优先使用旧 `last_checked_at`。
- 可用转待检时同时更新状态变更时间。
- 维护任务改为低频周期，并避开正在运行的拉取/检测任务。
- 自动检测优先于自动拉取，降低低库存拉取长期压制检测的风险。
- 只有自动任务相关设置变化时才调整对应调度时间；分页、导出目标和网关设置不得推迟自动任务。

## 3. 数据库兼容方案

新增字段：

```sql
ALTER TABLE proxies ADD COLUMN status_changed_at TEXT;
ALTER TABLE proxy_checks ADD COLUMN status_changed_at TEXT;
```

兼容原则：

- 启动时自动补列并回填现有记录。
- 不删除现有字段，不要求用户手动迁移。
- 旧版本数据库可以直接升级，回滚时新增列不会影响旧二进制读取。
- 迁移只修正状态语义，不删除代理。

## 4. 验收测试

必须覆盖：

1. 命名目标仅基础出口可达时状态为失败，但能力标记仍为 `base`。
2. 一个代理 `generic` 可用、`openai` 失败时，全局聚合状态保持可用。
3. 多目标任务中单目标失败不会立即删除代理。
4. 所有计划目标失败且基础出口不可达时，“失败即删”才会删除。
5. 取消请求后，新重任务仍被阻止，直到旧 worker 确认取消。
6. 全部代理源失败时任务终态为失败；部分失败时为部分成功。
7. 检测结果落库失败时任务不得显示完全成功。
8. 可用转待检后，待检 TTL 从状态转换时重新计算。
9. 保存导出目标等无关设置不得重置自动检测时间。
10. 启动迁移能修正旧 base-only 目标可用记录。
